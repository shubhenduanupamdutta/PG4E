# Notes on Inverted Indexes In PostgreSQL

---

## Inverted Indexes In PostgreSQL

---

**There are two main types of inverted indexes in PostgreSQL:**

- **GIN (Generalized Inverted Index)**
- **GiST (Generalized Search Tree)**

### Notes

- Like the B-Tree, the `GIN` is the usual _go-to_ inverted index and `GiST` is used in more special case. Previous manual implementation was a rough approximation of a `GIN` index.
- Both `GIN` and `GiST` want to know something about the type of array data it will be indexing and the kinds of operations that we will be using in `WHERE` clauses.

### Common Operators for Array Types

- `<@` (is contained by) - `SELECT * FROM table WHERE column <@ ARRAY['word1', 'word2'];` This operator checks if the array in the column is contained within the specified array.
- `@>` (contains) - `SELECT * FROM table WHERE column @> ARRAY['word1', 'word2'];` This operator checks if the array in the column contains all elements of the specified array.
- `&&` (overlaps) - `SELECT * FROM table WHERE column && ARRAY['word1', 'word2'];` This operator checks if the array in the column has any elements in common with the specified array.

---

## Generalized Inverted Index (GIN)

---

_GIN indexes are the preferred text search index type._

### Advantages

- Exact Matches
- Efficient on lookup/search

### Disadvantages

- Can be costly when inserting or updating data because every new word is inserted somewhere in the index and can get large

---

## Generalized Search Tree (GiST)

---

_A GiST index is lossy, meaning that the index might produce false matches, and it is necessary to check the actual table row to eliminate such false matches._ (PostgreSQL does this automatically when needed.) Hashing is used to reduce the size and cost to update the `GiST`.

### Advantages of GiST

- Faster to update than `GIN`

### Disadvantages of GiST

- Can produce false matches, which means that it may require additional checks against the actual data to confirm matches, potentially leading to slower query performance in some cases.

## Creating a GIN Index

---

### Creating a table to apply inverse index on

```sql
CREATE TABLE docs (
    id SERIAL PRIMARY KEY,
    doc TEXT
);
```

### Creating the inverse index

```sql
CREATE INDEX gin1 ON docs USING gin(string_to_array(doc, ' ') array_ops);
```

- `array_ops` is used to specify that we want to use the array operator class for the index, which allows us to use operators like `<@`, `@>`, and `&&` in our queries.

After creation of the index, we can see the following details about the table and the index:

```sh
natural_language=# \d+ docs

    "docs_pkey" PRIMARY KEY, btree (id)
```

### NOTE: The `string_to_array(doc, ' ')` should match exactly with the expression used in the `WHERE` clause of the query for the index to be used. So expression in create index and query should be the same

### Data in `docs` table

```sh
natural_language=# SELECT * FROM docs ORDER BY id LIMIT 3;
 id |                         doc
----+-----------------------------------------------------
  1 | This is SQL and Python and other fun teaching stuff
  2 | More people should learn SQL from UMSI
  3 | UMSI also teaches Python and also SQL
(3 rows)

natural_language=# SELECT count(*) FROM docs;
 count
--------
 100003
(1 row)
```

Lot of the rows are data generated by using `INSERT INTO docs (doc) SELECT repeat('Shubh', 5) || generate_series(1, 100000);`

- This is because other wise we would only have 3 rows in the tables and PostgreSQL will optimize the query to a sequence scan, since in this case scanning 3 rows is faster than using the index. With 100,000 rows, PostgreSQL will use the index to speed up the query.
- One another thing is that if you quickly query after insert or update, sometime Postgres may not use the index because it has not yet updated the inverted index.

### Some queries using the GIN index

```sql
SELECT id, doc FROM docs WHERE '{learn}' <@ string_to_array(doc, ' ');
```

This query will return rows where the array created from the `doc` column contains the word 'learn'. The `<@` operator checks if the array on the left (the specified array) is contained within the array on the right (the array created from the `doc` column).

```sh
natural_language=# SELECT id, doc FROM docs WHERE '{learn}' <@ string_to_array(doc, ' ');
 id |                  doc
----+----------------------------------------
  2 | More people should learn SQL from UMSI
(1 row)
```
